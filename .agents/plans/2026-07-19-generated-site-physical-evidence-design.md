# Generated Site Physical Evidence Design

Status: approved

## Outcome

Seed `73`, stage `1` produces a deterministic settlement export whose proved physical
facts let the production Raids preflight return `ready-for-scenario` without Settlements
assigning arrival, extraction, objective, guard, loot, trap, or encounter roles.

If this existed, we would no longer need a hand-authored synthetic base to prove that a
generated site can supply raid-ready physical evidence.

## Requirements

- `BUILD-1`: every connection declares physical movement modes; the site has no global
  movement type.
- Emit only generated-site geometry facts and neutral physical tags with stable source
  IDs.
- Preserve deterministic ordering, canonical identity, JSON export, earlier-stage facts,
  and the Actor-free recipe boundary.
- Distinguish measured evidence from unknown evidence at recipe level, including a
  measured category whose resulting array is empty.
- Do not modify the Raids-owned preflight, tactical mapping, scenarios, policies, or
  scores.
- Do not add aquatic generation, runtime growth behavior, final assets, or shared Unreal
  integration.

Requirements checked: Global (none recorded), BUILD-1; exceptions: none. Generated
content and Raids ownership conventions apply.

## Chosen machine

Enrich the existing building assembly and settlement recipe adapter. Do not create a
parallel challenge generator.

| Node | Edge/source | Contract | Work deleted | Independent verifier |
|---|---|---|---|---|
| Building usable-volume source | Adapt existing assembly | Each accepted assembly exposes stable ground-floor usable volume and doorway-opening clearance derived from its massing | Adapter guesses about interiors and doors | Building evaluator plus opening/volume counterexamples |
| Settlement evidence compiler | Adapt existing site adapter | Coarse paths and assemblies become a fine physical graph with modes, clearance, slots, visibility, blockers, coverage, and provenance | Hand-authored tactical geometry | Built Site recipe/adapter automation |
| Built Site recipe and JSON | Adapt existing contract/exporter | New facts participate in validation, checksum identity, ordered JSON, and atomic export | Second interchange format | Serializer tests and standard JSON parsing |
| Raids readiness preflight | Borrow | Production export is accepted as `ready-for-scenario`; removing each required category fails causally | Synthetic readiness claim | Raids-owned Python harness |

Sequence: building evidence → settlement evidence graph → validated recipe → JSON export
→ Raids preflight.

## Physical source facts

`FGatersBuildingAssembly` gains two small pure-data outputs:

- one stable ground-floor usable space with center and positive extent inside the walls;
- one stable entrance opening with transform, positive width and headroom, and the source
  door-module ID.

The existing module span, wall thickness, floor height, massing, and door-frame shape
produce these facts. `FGatersBuildingEvaluator` independently rejects missing,
non-finite, non-positive, out-of-envelope, or source-disconnected facts.

## Built Site contract

- `FGatersBuiltSiteConnection` gains ordered `MovementModeIds`; the current land adapter
  emits physical mode `ground` on every connection. JSON uses `movementModeIds`.
- `FGatersBuiltSiteRecipe` gains `FGatersBuiltSiteEvidenceCoverage`, containing placement,
  traversal-clearance, visibility, and blocker booleans plus stable source IDs. Coverage
  means all geometry generated by this site recipe was measured for that category; it
  does not claim environmental or runtime obstacle coverage. JSON uses
  `evidenceCoverage`.
- Coverage and movement modes participate in canonical text, checksum, validation, and
  deterministic JSON.
- Validation requires non-empty unique movement modes on every connection. When traversal
  coverage is declared, every connection requires positive width and headroom.
- Existing zero clearance and empty arrays remain valid unknown evidence when their
  coverage category is not declared.

The contract and export versions remain `1`; the green-field schema changes in place.

## Evidence compilation

- `FGatersSettlementEvidenceSettings` explicitly supplies the versioned physical mode,
  reserved path width/headroom, maximum evidence-graph link length, and neutral slot
  clearance. These are construction/evidence settings, not agent, scenario, policy, or
  balance facts; values live in code/data rather than prose.
- Reserved path cells become positive outdoor corridor volumes rather than centerlines.
- Coarse path adjacencies are deterministically subdivided so every connection fits the
  current Combat `ground-baseline` move-distance envelope. The independent acceptance
  test performs that comparison; the generator consumes only its physical evidence
  settings.
- Outdoor headroom is bounded by the generated-site evidence envelope; building modules
  are checked not to intrude into it.
- Building spaces use the proved ground-floor usable volumes, not whole building
  envelopes.
- Door connections use the proved opening width/headroom. Step height is the measured
  difference between connected floor surfaces; zero jump distance means no jump is
  required.
- Every positive-volume corridor and interior may emit one center placement slot. Tags
  describe only geometry such as `outdoors`, `settlement-edge`, `indoors`, `room`, or
  `doorway-adjacent`.
- Solid wall modules become physical blockers. The doorway module becomes frame blockers
  around its proved opening, never a solid blocker across the opening.
- Each open directed connection gains a directed visibility fact through the same proved
  corridor or opening. Distances and sample heights come from the physical endpoints;
  blocker references remain empty only after generated geometry was checked.

Stable IDs derive from existing space, path, building, module, and subdivision IDs.
Earlier settlement stages must retain byte-equivalent facts for every retained ID.

## Failure behavior

- Invalid building usable volume or opening rejects that assembly causally.
- Any unproved corridor, opening, clearance, containment, visibility line, blocker
  reference, movement mode, or coverage declaration rejects compilation without a
  partial accepted recipe.
- Removing placement, visibility, or blocker evidence, or clearing any required
  width/headroom, returns the matching Raids `*-evidence-unknown` result.

## Evidence gate

- Observe TDD RED before each production behavior.
- Focused building, recipe, adapter, JSON, and Built Sites automation pass.
- Repeated seed `73`, stage `1` exports are byte-identical and standard JSON retains all
  new facts and source IDs.
- The exact RAID-2 command regenerates the held-out artifact.
- `python research/raids-dungeons/rift_raid_harness.py --built-site-export <artifact> --summary`
  returns `ready-for-scenario`.
- One-category-at-a-time mutations return the four causal unknown-evidence findings.
- Complete `Gaters` automation, shared-agent docs, machine registry, and `git diff
  --check` pass.

## Rejected

- Separate challenge-base generator: duplicates working settlement machinery.
- Hand-authored ready fixture: does not prove generated capability.
- Tactical slot tags: violate the Raids ownership boundary.
- Site-wide movement type: violates BUILD-1 and blocks mixed networks.
- Full aquatic generator now: does not help the nearest RAID-3 evidence frontier.
